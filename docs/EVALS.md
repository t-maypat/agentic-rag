# Evaluation

Loupe is built around a thesis — *auditable* answers, not just answers — so
evaluation is a first-class deliverable, not an afterthought. There are two layers:

1. A **deterministic retrieval eval** that runs offline in CI on every PR and gates
   merges against a committed baseline.
2. A **judged generation eval** that runs the full agent nightly (or on manual
   dispatch) against live APIs and is reviewed by a human — never a merge gate.

Both reuse the same pure, unit-tested pieces (`app/eval/`) that the CI job runs, so
there is one source of truth for metrics.

---

## Datasets (`data/eval/`)

Both datasets are hand-curated against the committed corpus
(`data/corpus/ai_research_corpus.json` → `data/index/chunks.jsonl`, 92 chunks).

### `golden_retrieval.jsonl` — 40 items

`{qid, question, relevant_chunk_ids: [...], category}`

| category | count | notes |
|---|---:|---|
| definitional | 10 | "What is X?" → the doc's summary chunk |
| comparative | 10 | "How does X differ from Y?" → chunks from both |
| specific-fact | 10 | a precise detail (param counts, objectives, metric names) |
| multi-hop | 5 | needs chunks from 2–3 documents |
| unanswerable-from-corpus | 5 | `relevant_chunk_ids: []` — out of domain |

`relevant_chunk_ids` reference real ids in `chunks.jsonl` (e.g. `bm25:0`).

### `golden_generation.jsonl` — 25 items

`{qid, question, reference_answer, key_facts: [...], answerable}` — 20 answerable +
5 unanswerable.

## 1. Deterministic retrieval eval (CI gate)

**Script:** `scripts/run_retrieval_eval.py` · **CI job:** `eval-retrieval` in
`.github/workflows/ci.yml`.

Runs the **full hybrid pipeline** — `LocalNumpyStore` (dense cosine over committed
embeddings) + real BM25 + the production `app.retrieval.fusion.fuse` — with **zero
network and zero LLM calls**. Metrics are computed over the **35 answerable items**:

- `recall@5`, `recall@10`, `mrr@10`, `ndcg@10`, overall and per category.

Candidates are retrieved to depth 20 (`harness.CANDIDATE_K`) before the @5/@10
cutoffs so the metrics are well-defined after fusion dedup. `HYBRID_ALPHA` matches
the server default (0.6).

### Why the 5 unanswerable items are excluded from the gate

Min-max score normalization makes the top *fused* score ≈ 1.0 for **every** query,
so no fused-score threshold can detect "nothing relevant." For unanswerable items
the eval reports the top **raw dense cosine** (informational only). Actual refusal
correctness is measured by the generation eval below, where the *grade node* — not a
score threshold — makes the call.

### The gate

Fails CI when `recall@5 < baselines["recall@5"] - 0.02`. `data/eval/baselines.json`
is a committed ratchet, bumped deliberately in PRs that improve retrieval:

```bash
uv run python scripts/run_retrieval_eval.py                 # evaluate + gate
uv run python scripts/run_retrieval_eval.py --update-baselines   # bump the ratchet
```

### Fixtures & drift guard

Embeddings are committed under `data/eval/fixtures/` so CI never needs a key or the
network:

- `corpus_embeddings.npz` — every chunk embedded with `gemini-embedding-001` @ 768.
- `query_embeddings.npz` — the 40 golden-retrieval questions embedded.

Both carry metadata (`corpus_version` = sha256[:16] of `chunks.jsonl`;
`questions_signature` = hash of the golden question texts). The eval asserts both
match before running and **fails loudly with regeneration instructions** if the
corpus or questions drifted out of sync (`app/eval/fixtures.py`). Regenerate only
when the corpus or golden questions change:

```bash
uv run python scripts/build_eval_fixtures.py   # requires a valid GEMINI_API_KEY
```

## 2. Judged generation eval (nightly + manual)

**Script:** `scripts/run_generation_eval.py` · **Workflow:** `nightly-eval.yml`
(03:00 UTC + `workflow_dispatch`).

Runs the full graph in **quick mode** (real Pinecone + Gemini) over
`golden_generation.jsonl`, then a **separate judge pass** (`gemini-2.5-flash`,
temp 0, structured output) scores each item:

- **faithfulness** (1–5): are the answer's claims supported by the *retrieved*
  evidence?
- **key-fact coverage**: fraction of the item's `key_facts` conveyed by the answer.
- **correct-refusal** (bool): for the 5 unanswerable items, did the agent decline
  instead of fabricating?

Aggregates print to stdout; the full per-item report is written to
`eval_reports/<date>.json` (gitignored) and uploaded as a workflow artifact.

**Budget:** the run aborts after **150 LLM calls** total (generator + judge).

**Not a merge gate.** It is nondeterministic, and the judge shares a vendor with the
generator — a known limitation. Mitigations: the adapter seam (`LLMClient`) lets the
judge be swapped for another vendor, and the deterministic retrieval gate above is
the primary regression net. Regressions here are reviewed by a human from the
artifact.

Nightly requires `GEMINI_API_KEY` and `PINECONE_API_KEY` repository secrets.

## What runs where

| | PR CI (`ci.yml`) | Nightly / manual (`nightly-eval.yml`) |
|---|---|---|
| network / keys | none | live Gemini + Pinecone |
| retrieval eval (gate) | ✅ | — |
| generation eval (judged) | — | ✅ (budget-capped, human-reviewed) |

Metric definitions live in `app/eval/metrics.py` and are unit-tested in
`tests/unit/test_eval_metrics.py` (offline, using a synthetic embedding matrix).
